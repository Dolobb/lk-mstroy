# Vehicle Status — История и Возможности

## Что реализовано

- Синхронизация с Google Drive (нативный `.xlsx` через Drive API + SheetJS)
- Авторизация через Service Account (JWT, `creds.json`)
- Парсинг 8 вкладок Excel с автоопределением строки заголовков (поиск по первым 30 строкам)
- Idempotent upsert: повторный запуск sync в тот же день не создаёт дубликатов
- История ремонтов в PostgreSQL: каждый период ремонта = отдельная запись
- Автоматическое закрытие записи (установка `date_end`) при восстановлении ТС
- Счётчик `days_in_repair` обновляется при каждом sync
- HTTP API (Express :3004): список записей с фильтрами, запуск sync, статус sync
- Фронтенд (React): таблица с фильтром по статусу и категории, кнопка синхронизации
- Polling: после нажатия кнопки UI ждёт завершения sync (проверяет каждые 2 секунды)
- Safety timeout: polling останавливается через 60 секунд в любом случае
- Отображение ошибок синхронизации (до 3 штук + счётчик остальных)
- Логика isBroken: «требует ремонта» = исправен (не фиксируется как поломка)

## Что можно получить из текущей архитектуры

| Возможность | Сложность | Описание |
|-------------|-----------|---------|
| Автосинхронизация по расписанию | Низкая | Добавить `node-cron` в `index.ts`: `cron.schedule('0 9 * * *', runSync)` |
| Сортировка по любой колонке в UI | Низкая | Добавить `ORDER BY` параметр в `queryAll` и кнопки сортировки в таблицу |
| Экспорт таблицы в CSV | Низкая | Добавить `GET /api/vs/vehicle-status/export` — `queryAll` + csv-stringify |
| Пагинация | Низкая | Добавить `LIMIT`/`OFFSET` в `queryAll`, `?page=&limit=` в API |
| Сохранение состояния sync в БД | Средняя | Добавить таблицу `vehicle_status.sync_log`, писать результат sync в неё |
| Поиск по гос. номеру | Низкая | Добавить фильтр `?plate=` в API и поле поиска в UI |
| Уведомления о новых поломках | Средняя | Телеграм-бот или email: сравнить состояние до/после sync, отправить diff |
| Дашборд: ТОП машин по количеству ремонтов | Низкая | SQL-агрегация по `plate_number`, COUNT записей, средний `days_in_repair` |
| График ремонтов по времени | Средняя | Агрегация по `date_start` (по неделям/месяцам) + chart-компонент |
| Фильтр по диапазону дат | Низкая | Добавить `?dateFrom=&dateTo=` в `queryAll` |
| Webhook при завершении sync | Средняя | `POST` на внешний URL после `runSync()` завершится |
| Несколько файлов Drive | Средняя | Параметризовать `GOOGLE_SHEET_ID` или добавить массив в конфиг |

## Что ограничено архитектурой

- **Нет ретроспективы** — Drive хранит только 3 ревизии файла за всё время (март 2024, июнь 2024,
  текущее состояние). Импортировать историческое состояние невозможно.
- **Нет автосинхронизации** — по умолчанию sync запускается только вручную через UI или `POST /api/vs/vehicle-status/sync`.
- **Состояние sync в памяти** — `lastSync` / `lastResult` / `inProgress` сбрасываются при рестарте
  сервиса. После рестарта UI покажет «—» для времени последней синхронизации.
- **Файл скачивается целиком** при каждом sync. Инкрементальное обновление невозможно
  (Drive API не поддерживает diff для `.xlsx`).
- **Нет WebSocket** — polling каждые 2 секунды. При большом числе одновременных пользователей
  создаёт лишнюю нагрузку.
- **Один сервисный аккаунт** — если файл переместят или изменят права — sync сломается.
- **Жёсткая привязка к структуре Excel** — изменение названия вкладки или колонок без правки
  кода сервиса приведёт к ошибкам sync.
- **Нет дедупликации по `plate_number` внутри одного sync** — если одно гос. номер встречается
  в двух вкладках, оба будут обработаны независимо (разные `category`).
