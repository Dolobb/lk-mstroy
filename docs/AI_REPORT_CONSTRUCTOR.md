# AI-конструктор отчётов — концепция

## Проблема

Отчёты — основная работа. Данные лежат в разных базах и системах, каждый отчёт требует ручной компоновки: открыть одну таблицу, скопировать оттуда, открыть другую, сопоставить, оформить в Excel. На каждый новый формат отчёта — часы ручной работы.

## Решение

Встроенный чат в ЛК Мстрой. Пишешь текстом какой отчёт нужен — система сама достаёт данные из нужных баз, компонует и выдаёт готовый XLSX.

```
┌─────────────────────────────────────────────────────┐
│  💬 Какой отчёт нужен?                              │
│                                                     │
│  > Сводка КИП по всем тягачам СМУ-3 за февраль,     │
│    с разбивкой по сменам                            │
│                                                     │
│  ✅ Найдено 24 ТС, 312 смен                         │
│  📥 [Скачать XLSX]                                  │
└─────────────────────────────────────────────────────┘
```

Под капотом — нейросеть Claude Haiku (Anthropic). Она не считает сама — она понимает запрос и вызывает готовые функции для выгрузки данных из наших баз.

## Какие данные уже есть

| Источник | База | Что хранится |
|----------|------|-------------|
| КИП техники | PostgreSQL 16 (`kip_vehicles`) | КИП%, загрузка%, расход топлива, моточасы, простои, GPS-треки, геозоны по сменам |
| Тягачи | SQLite (`archive.db`) | Заявки, путевые листы, маршруты, пробег, стоянки, мониторинг по ТС |
| Самосвалы | PostgreSQL 17 (`mstroy`) | Рейсы (погрузка→выгрузка), КИП смены, время на объекте, события зон |
| Ремонт (из диспозиции) | PostgreSQL 17 (`repairs`) | Время ремонта, статус |
| Гео-данные | PostgreSQL 17 (`geo`) | Объекты, зоны погрузки/выгрузки, границы площадок |

## Примеры использования

### 1. Еженедельная сводка КИП

**Запрос:** «Сводка КИП по всей технике за прошлую неделю, сгруппировать по филиалам»

**Что сделает система:**
- Выгрузит `vehicle_records` за период (КИП%, загрузка%, моточасы, простои)
- Обогатит данными из реестра ТС (филиал, тип техники)
- Сгруппирует по филиалам, посчитает средние значения
- Выведет XLSX с листами по филиалам

### 2. Анализ расхода топлива

**Запрос:** «Покажи ТС с расходом топлива выше нормы за февраль»

**Что сделает система:**
- Возьмёт `fuel_rate_fact` и `fuel_rate_norm` из `vehicle_records`
- Отфильтрует где `fuel_variance > 1` (факт выше нормы)
- Выведет таблицу: госномер, модель, норма, факт, превышение в %

### 3. Отчёт по рейсам самосвалов

**Запрос:** «Рейсы самосвалов на объекте Тобольск за эту неделю с временем погрузки и выгрузки»

**Что сделает система:**
- Из `dump_trucks.trips` — все рейсы с зонами погрузки/выгрузки и длительностью
- Из `dump_trucks.shift_records` — привязка к ТС и смене
- Из `dump_trucks.zone_events` — детализация по времени в каждой зоне
- XLSX: дата, смена, госномер, зона погрузки, время погрузки, зона выгрузки, время выгрузки, длительность рейса

### 4. Простои техники

**Запрос:** «Какая техника простаивала больше 4 часов за смену на прошлой неделе?»

**Что сделает система:**
- Из КИП: `idle_time > 4` часа в `vehicle_records`
- Из самосвалов: `engine_time_sec` vs длительность смены в `shift_records`
- Объединит в одну таблицу с указанием типа ТС и подразделения

### 5. Кросс-отчёт: заявки + фактическое выполнение

**Запрос:** «По заявкам за январь покажи: сколько ПЛ выписано, сколько ТС работало, фактический пробег»

**Что сделает система:**
- Из тягачей: `tracked_requests` + `pl_records` — заявки и привязанные ПЛ
- Из мониторинга: фактический пробег и время работы
- Сводка: номер заявки, маршрут, кол-во ПЛ, кол-во ТС, суммарный пробег

### 6. Техника в ремонте vs на линии

**Запрос:** «Сколько самосвалов на объекте Тобольск было в ремонте, сколько работало — по дням за февраль»

**Что сделает система:**
- Из `dump_trucks.repairs` — ТС в ремонте по датам
- Из `dump_trucks.shift_records` — ТС которые работали
- Таблица по дням: дата, на линии, в ремонте, процент доступности

### 7. Нехватка данных — система подскажет

**Запрос:** «Отчёт по расходу ГСМ с учётом заправок на АЗС»

**Ответ системы:**
> Данные по расходу топлива (факт и норма) доступны из КИП.
> Данные по заправкам на АЗС отсутствуют — в системе нет источника с чеками АЗС.
> Для этого отчёта нужно добавить таблицу с данными АЗС.
> Могу сформировать отчёт без АЗС — только факт/норма из мониторинга.

## Как это наращивается

Каждый источник данных — это «функция» для нейросети. Сейчас их 4 (КИП, тягачи, самосвалы, гео). Добавление нового:

```
1. Появляется потребность в новом отчёте
2. Определяем каких данных не хватает
3. Создаём таблицу в БД + парсер для загрузки данных
4. Добавляем одну функцию (~30 строк) для AI
5. Готово — AI может комбинировать новый источник со всеми старыми
```

**Накопительный эффект:** 5 источников дают не 5, а десятки комбинаций для отчётов. Каждый новый источник делает систему экспоненциально полезнее.

---

## Безопасность

- Нейросеть **не имеет прямого доступа** к базам данных
- Она вызывает только заранее описанные функции (read-only выборки)
- Не может удалить, изменить или повредить данные
- Все запросы логируются
