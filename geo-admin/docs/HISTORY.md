# Geo Admin — История и Возможности

## Что реализовано

- Express + PostGIS backend на порту :3003
- Leaflet-карта с отображением объектов и геозон
- Импорт геозон из `kip/config/geozones.geojson` (291 зона, 282 объекта, выполнен 2026-02-18)
- CRUD объектов: создание, просмотр, редактирование (name/smu/region), удаление
- CRUD зон: создание через рисование на карте, редактирование (name/tags), удаление
- Теги зон: 5 типов (`dt_boundary`, `dt_loading`, `dt_unloading`, `dt_onsite`, `dst_zone`)
- Фильтрация зон на карте: «ДСТ» (dst_zone + dt_boundary) / «Самосвалы» (dt_loading + dt_unloading + dt_boundary)
- Цветовое кодирование зон по типу тега
- Рисование полигонов через leaflet-draw
- Модальные формы создания/редактирования
- Accordion-список объектов в сайдбаре с количеством зон
- Zoom на зону при выборе из списка
- Каскадное удаление зон при удалении объекта (ON DELETE CASCADE)
- GIST-индекс на geometry для пространственных запросов
- Миграционная система (SQL-файлы + tracking table `geo._migrations`)
- Slug-генерация uid из названия объекта (транслитерация кириллицы)
- Дедупликация uid при коллизии (суффикс -2, -3, ...)
- Транзакционные операции создания/обновления зон (zones + zone_tags атомарно)
- Валидация тегов на стороне API (whitelist из 5 значений)
- Error-баннер на фронтенде (5 сек) при сбоях API
- Health endpoint `/api/geo/health`

## Что можно получить из текущей архитектуры

### Простые доработки (1–4 часа)

- **Поиск объектов по названию** — фильтр в сайдбаре, дополнительный GET-параметр `?search=` в `/api/geo/objects`
- **Экспорт геозон в GeoJSON** — новый endpoint `GET /api/geo/export`, возвращает FeatureCollection всех зон
- **Редактирование геометрии зоны** — leaflet-draw поддерживает `L.Draw.Edit`, нужно подключить и передать новую геометрию в `PUT /api/geo/zones/:uid`
- **Показ объекта на карте всегда** — сейчас при переключении фильтра зоны объекта, не имеющего нужных тегов, исчезают с карты; добавить «всегда показывать все» режим
- **Копирование uid зоны** — кнопка в popup или в сайдбаре для передачи uid в другие системы

### Средние доработки (4–16 часов)

- **Пространственный поиск** — `ST_Intersects(geom, ST_Buffer(ST_Point($1,$2)::geography, $3)::geometry)` — найти зоны в радиусе точки
- **Интеграция в единый `frontend/`** — сейчас geo-admin живёт отдельно; можно встроить как iframe или переписать на React с тем же API
- **Pagination объектов** — при 282+ объектах сайдбар перегружен; добавить `LIMIT/OFFSET` в API и виртуальный скролл
- **Группировка объектов по smu/region** — сортировка и группы в сайдбаре
- **Привязка зон к ТС** — новая таблица `geo.zone_vehicle_types`, API для получения зон по типу ТС

### Сложные доработки (требуют исследования)

- **Редактирование геометрии в UI** — leaflet-draw `EditToolbar` требует работы с `FeatureGroup`, сложнее чем рисование
- **Версионирование геозон** — таблица `geo.zone_history` с хранением старых геометрий
- **Кластеризация на карте** — при большом количестве зон может понадобиться leaflet.markercluster или tile-based rendering

## Что ограничено архитектурой

- **Нет React** — фронтенд на Vanilla TS. Добавление сложных компонентов (date picker, multi-select с поиском) потребует либо подключения библиотек через CDN, либо переписывания на React
- **Нет авторизации** — все API endpoints открыты без auth. Подойдёт для внутренней сети, но не для публичного доступа
- **Нет WebSocket** — обновления в реальном времени не поддерживаются; при параллельной работе двух пользователей изменения не синхронизируются
- **Только полигоны** — `GEOMETRY(Polygon, 4326)`. Точки и линии не поддерживаются схемой БД
- **Геометрия только без дыр** — `Polygon` без MultiPolygon; зоны с «дырами» не представимы
- **Один тег-приоритет для цвета** — цвет зоны на карте определяется первым совпавшим тегом по приоритету; многотеговые зоны визуально неразличимы
- **leaflet-draw без редактирования** — только рисование новых; изменение формы существующей зоны через UI не реализовано
- **Нет full-text search** — поиск объектов и зон не реализован; для 282 объектов пока приемлемо
